# /work-plan — 작업 계획 수립 (구현 없음)

**사용법**: `/work-plan "<작업 설명>"`
예) `/work-plan "중복 웹훅 방지 — Redis idempotency 체크 추가"`

> 코드를 작성하지 않는다. 계획만 수립하고 사용자 승인을 기다린다.

---

## 실행 절차

### 1. 현재 상태 파악 (직접 실행)

$ARGUMENTS를 분석해 관련 파일을 판단하여 Read한다. 가정 금지 — 반드시 실제 코드를 읽은 뒤 계획을 작성한다.

최소한 아래는 항상 확인한다:
- `app/` 내 영향받을 것으로 보이는 파일
- `tests/` 내 관련 테스트 파일
- `requirements.txt` (새 패키지 필요 여부)

### 1-1. 중단 조건 (파일 읽기 후 즉시 판단)

아래 상황이면 계획 수립을 중단하고 사유를 출력한다. 계획 항목을 채우지 않는다.

| 상황 | 중단 사유 출력 |
|---|---|
| $ARGUMENTS가 모호해서 영향 파일을 특정할 수 없음 | "작업 설명이 불명확합니다 — 어떤 파일의 어떤 동작을 바꾸는지 구체적으로 설명해 주세요." |
| 영향 범위가 3개 이상의 커맨드를 병행해야 하는 복잡도 | "작업 범위가 너무 넓습니다 — 다음과 같이 분리를 제안합니다: [분리 목록]" |
| 필요한 핵심 파일이 존재하지 않음 | "필요한 파일이 없습니다: [파일 경로]" |

---

### 2. 계획 출력

아래 항목을 순서대로 출력한다.

---

#### 작업 요약
- 한 줄로: 무엇을 왜 바꾸는가

#### 수용 기준 (Acceptance Criteria)
완료로 볼 수 있는 조건을 구체적으로 명시한다. 모호한 표현("잘 동작한다") 금지.
- [ ] 조건 1 (예: 중복 웹훅 수신 시 두 번째 요청이 202를 반환하되 DB에 저장되지 않는다)
- [ ] 조건 2 (예: 기존 테스트가 모두 통과한다)
- [ ] 조건 3 (예: ruff lint 에러 0건)

#### 사용할 커맨드
- `/feature-modify`, `/migration-add`, `/provider-add`, `/hotfix` 중 어느 것을 쓸지, 왜인지

#### 영향 범위
수정이 필요한 파일 목록 (실제 Read한 코드 기준):
- `app/xxx.py` — 변경 이유
- `tests/xxx.py` — 변경 이유
- `requirements.txt` — 추가 패키지 필요 시

#### DB 스키마 변경 여부
- 필요 없음 / 필요함 (어떤 컬럼·테이블)
- 필요한 경우 `/migration-add` 병행 여부

#### 단계별 실행 계획

Phase 1 — [단계 이름]
- 작업: ...
- 검증:
  ```bash
  DYLD_LIBRARY_PATH=/opt/homebrew/opt/expat/lib \
    DATABASE_URL=postgresql+psycopg2://user:password@localhost:5433/webhook_db \
    .venv/bin/pytest tests/ -k "..." -v
  ```

Phase 2 — [단계 이름]
- 작업: ...
- 검증: (각 Phase마다 실제 검증 대상 테스트를 구체적으로 명시)

#### CLAUDE.md 제약사항 체크
- 순환 임포트 위험 여부
- Prometheus 메트릭 위치 (metrics.py 외 정의 금지)
- HMAC 비교 방식 (해당 시)
- 테스트 패턴 (dependency_overrides / collect() 패턴 해당 여부)

#### 리스크
- 예상 문제와 대응 방법

#### 추정 범위
- 수정 파일 수, 신규 테스트 케이스 수

---

### 3. 승인 대기

계획 출력 후 반드시 멈춘다. 코드 작성·파일 수정·브랜치 생성을 하지 않는다.

계획 검토 후 진행하려면 해당 커맨드를 실행해 주세요.
수정이 필요하면 피드백을 주세요.
